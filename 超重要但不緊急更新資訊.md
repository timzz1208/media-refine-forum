# 超重要但不緊急更新資訊

這份文件放「明天展示不一定要立刻處理，但正式公開前一定要回來處理」的事項。

## 1. Google Apps Script API 可能被濫用

### 為什麼重要

如果 Google Apps Script Web App 設定成：

```text
Who has access: Anyone
```

代表任何知道 `/exec` 網址的人，都可以嘗試送資料到 Google Sheet。

而且這個網址如果寫在前端程式裡，上架到 GitHub Pages 後，懂一點技術的人可以從瀏覽器開發者工具或原始碼看到。

### 可能風險

- 被亂送垃圾投稿。
- 被重複送大量資料，灌爆 Google Sheet。
- 有人偽造評分或投稿。
- 表格裡出現惡意連結或不適合內容。
- 如果未來接更多功能，可能變成資料污染入口。

### 現階段判斷

明天展示、小班內測可以接受。

但正式公開前，不能只靠公開 `/exec` 網址。

### 下一版最低防護

1. **加投稿 token**
   - 前端送資料時帶一組暗號。
   - Apps Script 檢查暗號正確才寫入。
   - 這不是完美安全，但能擋掉最基本的亂打。

2. **限制欄位長度**
   - 標題最多 100 字。
   - 原因最多 1000 字。
   - 摘要最多 3000 字。

3. **記錄來源與時間**
   - 每筆資料都記錄 timestamp。
   - 未來可加 userAgent 或來源頁面。

4. **分開投稿與評分權限**
   - 投稿可以比較開放。
   - 評分最好要知道評分者是誰。

5. **檔案上傳先不要走 Apps Script**
   - 圖片、PDF 先用 Google Form 檔案上傳。
   - Apps Script 先處理文字資料即可。

6. **管理者審核**
   - 新投稿先進待審核。
   - 審核後才進內容池或高分精修流程。

### 白話結論

現在的 Apps Script API 像是「收件信箱」。

明天展示可以用。

但正式公開前，需要至少加一把簡單的鎖，不能讓任何知道網址的人都能亂塞資料。

## 2. GitHub Pages 不是後端

GitHub Pages 只負責讓大家打開網頁。

真正把資料寫進 Google Sheet 的，是 Google Apps Script。

正式架構應該理解成：

```text
GitHub Pages：前台畫面
Google Apps Script：收件入口
Google Sheet：資料表
Google Drive / Google Form：檔案收件
```

## 3. localStorage 只能當備份

localStorage 的角色應該是：

- 使用者手機/電腦的暫存。
- 網路不穩時的備援。
- 展示原型資料。

它不應該被當成正式後台。

正式後台一定要有：

- Google Sheet
- Supabase
- Firebase
- 或其他真正資料庫

## 4. 正式公開前檢查清單

- Apps Script 加 token。
- Apps Script 限制欄位長度。
- Google Sheet 分投稿資料與評分紀錄。
- 評分者名稱要必填或登入。
- 投稿資料要有待審核狀態。
- 檔案上傳走 Google Form / Drive。
- README 標清楚這是 MVP，不是完整會員系統。
